# 计划：重新定义"资产/家庭/空间"概念，适配 AI 玩偶产品

## Context

当前文档体系中，设备归属使用了传统 IoT 的"资产 (Asset)"概念，采用"家庭 → 房间"树形结构。对于 AI 玩偶类产品，这个概念很不自然 —— 家长买了一个 AI 玩偶，不会想着"把它放到客厅这个资产节点下"。

**约束**：API 层面的字段名 (`assetId`, `bid`, `/asset/assetTree`) 是后端已有接口，文档无法改变接口本身。方案需要在不改 API 的前提下，让概念对 AI 玩偶客户合理。

## 问题分析

"资产树"在 AI 玩偶场景下的问题：
1. **概念不匹配**：玩偶不是"资产"，不属于"房间"
2. **不必要的复杂度**：玩偶产品通常不需要"家庭 → 房间"多级管理
3. **开发者困惑**：设备端/App 端开发者看到 `assetId` 不知道该传什么

## 方案：文档层面重新包装

### 核心思路

将"资产树"重新解释为**"账户空间"**的概念：
- `assetId` → 解释为**"用户账户 ID"**或**"设备归属 ID"**
- "资产树" → 解释为**"账户结构"**，对玩偶产品只用根节点即可
- 去掉所有"家庭/房间"的措辞，用**"账户"**或**"设备归属"**替代

### 具体做法

**对开发者文档**：在概念层面做映射，告诉开发者"这个字段在你的场景里代表什么"

| 原概念 | 新说法 | 开发者看到的 |
|--------|--------|-------------|
| 资产 (Asset) | 账户 (Account) | "设备绑定到用户账户" |
| 资产树 (Asset Tree) | 账户结构 | "获取用户账户 ID" |
| assetId | 账户 ID | "绑定时传入账户 ID（API 字段名为 assetId）" |
| bid | 账户 ID | "BLE 传递的账户 ID（字段名为 bid）" |
| 我的家 → 客厅 | （不提层级） | "直接使用根节点的 assetId" |

**对 API 参考文档**：保留字段名不变，但用注释说明含义

### 需要修改的文件

涉及面较大，按优先级排序：

**P0 — 核心概念文档**（改了这些，开发者的心智模型就对了）：
1. `architecture.md` — 第 3.5 节"资产 (Asset)"重写，第 4 节配网说明调整
2. `solution-ai-toy.md` — 配网流程、序列图中的措辞
3. `architecture-overview.md` — 销售版中的"多设备管理"描述

**P1 — 开发者指南**：
4. `guide-device.md` — 绑定流程中的 assetId 说明
5. `guide-app.md` — 资产树获取步骤重新包装
6. `quickstart-app.md` — "获取资产树"步骤重命名

**P2 — 参考文档**（字段名不改，注释调整）：
7. `ref-mqtt.md` — assetId 字段说明
8. `ref-ble.md` — bid 字段说明
9. `ref-rest-api.md` — 资产树接口描述

**不改**：
- `original-docs/` — 原始文档保留原样
- API 字段名 (`assetId`, `bid`, `/asset/assetTree`) — 这是后端接口，文档无法改变

- 确认开发者能理解 `assetId` 就是"账户 ID"，绑定时直接用根节点
- 确认 API 调用示例仍然正确可用
