## 一、为什么记忆系统是 AI 产品的核心问题

AI 应用上线时的表现往往令人惊艳，但用户使用三个月后会开始出现两类抱怨——"它怎么记得乱七八糟的东西"，以及"它把我跟别人搞混了"。这不是模型不够聪明的问题，而是记忆系统的设计问题。

绝大多数现有的记忆方案是"只进不出"——用户随口说的一句话和真正重要的事被同等保存，半年下来记忆库塞满噪音，AI 的回答质量随之下降。这是 mem0 等通用方案在生产环境最常被诟病的痛点（mem0 自身 GitHub Issue #4573 报告了一个生产实例：10,134 条记忆中 97.8% 是垃圾或重复）。

Sentino 在设计 Memex 框架时，把这个问题作为首要约束：**用户用得越久，AI 应该越懂他，而不是越乱**。

## 二、Sentino 记忆框架的三个产品价值

### 2.1 用得越久，AI 越懂用户

Sentino 的记忆系统给每条信息标记重要度，并按重要度决定保留时长——

- **核心信息**（用户身份、家庭关系、长期习惯、慢性病）：永久保留
- **重要信息**（长期偏好、进行中的项目、新建立的关系）：保留 30 天
- **普通信息**（日常事件、近期计划）：保留 14 天
- **琐碎信息**（今天的心情、刚才吃的什么、当下的措辞）：保留 3 天

过期的信息自动淡出，但底层数据可追溯（合规和审计需要时可恢复）。如果一件事被用户反复提起，系统会自动把它"续命"——重要的事自然活下来，没人再提的事自然淡出。

这套机制让记忆库长期保持精炼。Sentino 的判断是：**这跟人类记忆的运作方式是一致的**——人也不会平等地记住所有发生过的事。

### 2.2 成本是可预测的

mem0 等方案默认每收到一条用户消息就调用一次大模型做提取。对话量上去之后，token 成本随消息数量线性增长，对高频场景（语音陪伴、客服、咨询）这是月成本不可控的主要原因。

Sentino 采用**异步批处理**：用户消息先入库（毫秒级返回），等积累到一定量或定时触发时，一次大模型调用处理全部新消息。同等条件下，处理 19 条对话从 5 分钟降到 15 秒，**速度提升 19 倍，token 成本降至原方案的约 1/15**。

对客户而言：消息进入系统的延迟可以做到极低，记忆生成有可预测的批处理窗口，月成本可估算可控制。

### 2.3 区分"用户是谁"与"我们一起经历过什么"

mem0 等方案的记忆是一张扁平的事实列表，"用户住北京"和"用户上周去北京出差"以相似形式存在。Sentino 在数据层就把这两件事分开：

- **用户画像**（Profile）：用户的当前状态——属性快照，覆盖式更新
- **共同经历**（Event）：与用户的互动时间线——追加式记录，带衰减

这一区分对 AI 陪伴和对话型产品至关重要。用户期待的是两种不同的能力：

| 能力 | 撑起的产品体验 |
|---|---|
| **了解我是谁** | 个性化推荐、贴合用户身份的回应 |
| **记得我们经历过什么** | 情感连续性、关系深度感 |

混在一起做，两种体验都做不好。

## 三、技术路线：为什么不用向量检索

这是 Sentino 与 mem0 最根本的技术分歧，也是判断一个记忆系统是否成熟的关键信号。

### 3.1 向量检索的固有问题

向量检索是"语义相似度匹配"——把记忆和查询都转换为数学向量，按相似度排序取前几条。这是 2022-2023 年记忆系统的主流做法，问题在于：

- **会拉错东西**："我对花生过敏"和"我喜欢吃花生酱"的语义相似度很高，但意义完全相反。一旦 AI 用错记忆做判断，可能涉及健康、安全、合规风险。
- **不可审计**：当 AI 给出错误回答，工程师无法解释"为什么模型选了这条而不是那条"——向量空间是黑盒。
- **不可编辑**：用户要求"忘掉某件事"，向量库里没有"那件事"的精确位置可以删除。
- **不可预期**：同样的查询，因为底层向量索引的近似搜索特性，结果可能不一致。

### 3.2 Sentino 的选择：结构化文本记忆 + 大模型自判

Sentino 把记忆以结构化文本形式存储，检索时让大模型阅读相关全集自行判断哪些有用，**而不是用向量去"猜"哪条相关**。这意味着：

- 每条记忆人类可读、可编辑、可审计
- 用户要求修改或遗忘，可以精确定位
- 监管/合规审查可以追溯每一条 AI 决策的依据

### 3.3 这条路线的行业背书：Anthropic 同款判断

Anthropic 在 2025 年发布的 Claude Code（业内公认最先进的 AI 编程助手之一）使用的是同样的设计——**Markdown 文本文件 + 索引，不用向量数据库**。Anthropic 在工程文档中明确说明这一选择的理由：透明可编辑、原生支持 Git 版本控制、与文件系统天然集成。

行业的共识在 2025-2026 年发生转变：随着大模型上下文窗口扩大、推理成本下降，"让模型读全集自判"在大多数场景下比"向量检索预筛"更准确、更可解释、更易维护。

**Sentino 选这条路线，与 Anthropic 同方向。**

## 四、与 mem0 的对照（客户视角）

| 客户关注的问题 | mem0 | Sentino Memex |
|---|---|---|
| 三个月后记忆库还干净吗？ | 容易膨胀，需要人工运维 | 自动衰减，越用越精准 |
| 月度 token 成本如何控制？ | 随对话量线性增长 | 批处理，月度可预测 |
| AI 答错时能查原因吗？ | 向量检索黑盒，难定位 | 文本记忆可读可审计 |
| 用户要求"忘掉某事"能做到吗？ | 难精确定位 | 可定位到具体记忆条目 |
| "用户是谁"vs"经历过什么"分得清吗？ | 混在一起 | 数据层即分离 |
| 多语言对话支持？ | 取决于配置 | 内置多语言记忆，统一 prompt 架构优化缓存命中 |

## 五、适用边界

为了帮助客户判断是否合适，需要诚实说明 Sentino Memex 与 mem0 的定位差异：

**mem0 的优势场景**：通用 Agent 记忆基础设施，开源生态成熟，社区活跃，支持多种向量数据库适配。如果产品是工具型 AI（编程助手、研究助手、内部知识检索），对记忆精度容忍度较高，mem0 是合理选择。

**Sentino Memex 的优势场景**：**记忆质量直接决定用户留存的产品**——AI 陪伴、语音对话、客户咨询、关系型助手。这类场景对记忆精度、成本可预测性、长期不腐烂的要求高一个量级，需要为这类场景专门设计的取舍。

Sentino 不打算在通用 Agent 记忆赛道与 mem0 竞争。我们专注做"记忆是产品本身"的场景。

# 附录：技术规格

## A. 系统架构

```
对话输入 ──→ Blob 入库 (毫秒级) ──→ Flush 触发 ──→ AI 提取 ──→ 写入存储
   │                                                     │
   └─ HTTP API                                            ├─ User Profile (画像)
                                                         ├─ Memory Event (事件)
                                                         └─ Event Gist (可查询条目)
```

**部署形态**：Spring Boot 服务（JDK 17+，Maven 3+）
**存储后端**：PostgreSQL（主数据）+ Redis（缓存/锁）
**外部依赖**：兼容 OpenAI API 的大模型服务（GPT / Claude / Gemini / 国产模型均可）

## B. 数据模型

| 实体 | 用途 | 关键字段 |
|---|---|---|
| `Blob` | 原始对话输入 | userId, projectId, blobData, status, createdAt |
| `UserProfile` | 用户属性快照 | content, attributes(JSONB), updatedAt |
| `UserEvent` | 事件记录 | eventData(含事件描述+画像增量) |
| `UserEventGist` | 可检索的最小记忆单元 | gistData, importanceLevel, status, **forgetAt** |

多租户支持：所有数据按 `(userId, projectId)` 双键隔离。

## C. 重要度等级与 TTL

| 等级 | 保留时长 | 内容举例 |
|---|---|---|
| CORE | 永久 | 身份、家人、长期习惯（≥3 个月）、慢性疾病 |
| IMPORTANT | 30 天 | 长期偏好、进行中项目、新建立的关系 |
| NORMAL（默认） | 14 天 | 日常事件、周计划、一般陈述 |
| TRIVIAL | 3 天 | 当下情绪、今天吃了什么、含糊措辞 |
